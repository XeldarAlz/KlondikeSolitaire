## 1. Architecture Overview

### Pattern: Model-View-System (MVS)

```
┌─────────────────────────────────────────────────────────────────┐
│                        Views (MonoBehaviours)                    │
│  CardView · PileView · InputView · HudView · OverlayView       │
│  WinCascadeView · BoardView                                     │
│  ─ Observe Models via ReactiveProperty<T>.Subscribe()           │
│  ─ Call Systems for actions                                      │
│  ─ Own all animation (PrimeTween) and async timing (UniTask)    │
├────────────────────────┬────────────────────────────────────────┤
│                        │ MessagePipe                             │
│                        │ (cross-system & system→view events)     │
├────────────────────────┴────────────────────────────────────────┤
│                        Systems (Plain C#)                        │
│  DealSystem · MoveValidationSystem · MoveExecutionSystem        │
│  ScoringSystem · UndoSystem · HintSystem · AutoCompleteSystem   │
│  NoMovesDetectionSystem · GameFlowSystem                        │
│  ─ Own and mutate Models                                         │
│  ─ Contain ALL game logic                                        │
│  ─ No Unity API, no MonoBehaviour, no async                     │
├─────────────────────────────────────────────────────────────────┤
│                        Models (Pure C#)                          │
│  CardModel · PileModel · BoardModel · ScoreModel · GamePhaseModel│
│  ─ State + data only                                             │
│  ─ Zero dependencies                                             │
│  ─ Observable via ReactiveProperty<T>                            │
└─────────────────────────────────────────────────────────────────┘
```

### Key Architectural Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| DI Container | VContainer | Required by project constraints. Single LifetimeScope — no menus, no scene transitions |
| Messaging | MessagePipe | Required by project constraints. Cross-system events, readonly struct messages |
| Animation | PrimeTween | User-specified. Zero-allocation, struct-based tweens. Awaitable with UniTask |
| Async | UniTask | Required by project constraints. Views use for animation sequencing only |
| UI Framework | uGUI + TextMeshPro | Simple HUD (score + 4 buttons + 2 overlays). uGUI is proven and sufficient. UI Toolkit is overkill here and has PanelSettings foot-guns |
| Model observation | Minimal ReactiveProperty\<T\> | ~50 lines custom implementation. Avoids adding R3/UniRx for 5 observable properties. Views bind directly to model state |
| Card visuals | Editor-time composited single sprite | Each card = 1 SpriteRenderer. An Editor tool (`CardAtlasGenerator`) pre-composites card layers (front + rank + suit + figure) into 52 PNG files, packed into Unity's SpriteAtlas. Per-platform resolution via atlas overrides. Eliminates multi-sprite overdraw and enables full batching (all cards = 1 draw call) |
| Card flip animation | Scale-X squash/stretch | Clean for 2D, no perspective needed. PrimeTween scales X from 1→0 (swap sprites at midpoint) →0→1 |
| Undo | Command pattern (delta) | Stores move deltas, not snapshots. O(1) memory per step. Textbook fit for unlimited undo |
| Object lifecycle | Fixed 52 cards + stamp pool | Cards are pre-warmed once at app start, never destroyed. Moved between piles. Win cascade trail uses a stamp pool (~150, same SpriteAtlas = same batch) |
| Overdraw strategy | Strip sprites for face-down cards | Face-down cards in tableau render only the visible strip (top ~20%), not the full card back. Reduces stacked overdraw by ~80%. Full back only for top-of-pile. Strip sprite pre-generated by Editor tool |
| Systems purity | Zero Unity API in Systems | All systems are pure C#. No UniTask, no MonoBehaviour, no UnityEngine. Fully unit-testable without Unity |

## 2. Technical Constraints & Standards

All constraints from `CLAUDE.md` and `.claude/rules/` apply. Project-specific additions:

| Constraint | Detail |
|------------|--------|
| PrimeTween only | No DOTween, no coroutines, no manual Lerp in Update. All animation through PrimeTween |
| uGUI only | No UI Toolkit. TextMeshPro for all text |
| No persistence | No save/load, no PlayerPrefs, no file I/O. Score exists in memory only |
| No audio | No AudioSource, no AudioClip, no audio system |
| Single scene | Everything in one scene. No additive loading, no scene transitions |
| Desktop + Mobile | Landscape orientation, any resolution. Mouse and touch input via Input System Pointer abstraction. No gamepad |
| Draw 1 mode only | Single card draw from stock. No Draw 3 variant |

## 3. System Inventory

### Infrastructure

| System | Purpose |
|--------|---------|
| ReactiveProperty\<T\> | Minimal observable property for model→view binding |
| CompositeDisposable | Subscription lifecycle manager for views |
| GameLifetimeScope | VContainer LifetimeScope — wires all dependencies |

### Game Logic (Pure C#)

| System | Purpose |
|--------|---------|
| DealSystem | Fisher-Yates shuffle, deal 28 cards to tableau + 24 to stock |
| MoveValidationSystem | Validate any proposed card move against Klondike rules |
| MoveExecutionSystem | Execute validated moves, update board state, publish events |
| ScoringSystem | Track score, apply points per move type, clamp to 0 |
| UndoSystem | Command stack — record, reverse, and clear move history |
| MoveEnumerator | Shared stateless utility — enumerate all valid moves or short-circuit on first. Used by HintSystem and NoMovesDetectionSystem |
| HintSystem | Cache valid moves via MoveEnumerator, cycle through on request, reset on board change |
| AutoCompleteSystem | Detect trivially-winnable state, generate optimal move sequence |
| NoMovesDetectionSystem | After each move, check via MoveEnumerator.HasAnyValidMove if any valid moves remain |
| GameFlowSystem | State machine: Dealing → Playing → AutoCompleting → Won / NoMoves |

### Views (MonoBehaviour)

| View | Purpose |
|------|---------|
| CardView | Render one card (single SpriteRenderer, pre-composited sprite from SpriteAtlas), manage flip and sorting |
| PileView | Position cards within a pile, manage card offsets and layout |
| BoardView | Pre-warm 52 card GameObjects, assign to PileViews, manage dealing animation |
| InputView | Pointer input (mouse + touch via Input System Pointer) → system calls. Drag/tap/double-tap detection. Coordinates with DragView for drag lifecycle |
| DragView | Manage visual state of dragged cards (position tracking, sorting, snap-back). Stateless between drags |
| HudView | Score display, Undo/Hint/NewGame/AutoComplete buttons |
| OverlayView | Win screen and No-Moves prompt overlays |
| WinCascadeView | Cascading card bounce animation with trail effect |

## 4. Bootstrapping & Lifecycle

### Initialization Order

```
1. GameLifetimeScope.Configure()
   ├── Register Models (Singleton): BoardModel, ScoreModel, GamePhaseModel
   ├── Register Systems (Singleton): DealSystem, MoveValidation, MoveExecution,
   │   Scoring, Undo, Hint, AutoComplete, NoMovesDetection, GameFlow
   ├── Register MessagePipe brokers for all message types
   └── Register Views: find in hierarchy (CardView[], PileView[], InputView, etc.)

2. VContainer resolves all dependencies via constructor injection (Systems)
   and [Inject] Construct (Views)

3. BoardView.Start()
   ├── Pre-warm 52 CardView instances from Card prefab (1 SpriteRenderer each)
   └── Assign each CardView a face sprite + back sprite from CardSpriteMapping SO
       (sprites come from Cards.spriteatlas — pre-composited by Editor tool, no runtime cost)

4. GameFlowSystem receives StartNewGame call
   ├── DealSystem shuffles and assigns cards to PileModels
   ├── ScoreModel resets to 0
   ├── UndoSystem clears stack
   └── Publishes DealCompletedMessage

5. BoardView receives DealCompletedMessage
   ├── Assigns CardViews to PileViews based on model state
   └── Animates dealing sequence (PrimeTween + UniTask)

6. InputView enables after deal animation completes
   └── Gameplay begins
```

### New Game Flow

```
User clicks New Game → HudView calls GameFlowSystem.StartNewGame()
  → GameFlowSystem resets all systems (deal, score, undo, hints)
  → Publishes NewGameStartedMessage
  → BoardView receives message, reclaims all CardViews, animates new deal
  → InputView re-enables after deal animation
```

### Shutdown

No explicit shutdown needed — single scene, no persistence, no cleanup beyond standard Unity destruction. Systems implement `IDisposable` to unsubscribe from MessagePipe.

## 5. Assembly Definitions

```
KlondikeSolitaire.Core          (Assets/Scripts/Core/)
├── Models, enums, message structs, ReactiveProperty<T>, interfaces
├── References: NONE
└── Enforces: zero Unity dependency in domain layer

KlondikeSolitaire.Systems       (Assets/Scripts/Systems/)
├── All pure C# game systems
├── References: Core, MessagePipe
└── Enforces: no UnityEngine, no Views, no async

KlondikeSolitaire.Views         (Assets/Scripts/Views/)
├── All MonoBehaviours, LifetimeScope, ScriptableObjects
├── References: Core, Systems, VContainer, MessagePipe, MessagePipe.VContainer,
│   PrimeTween, UniTask, Unity.InputSystem, Unity.TextMeshPro, UnityEngine
└── Enforces: views can depend on everything downstream

KlondikeSolitaire.Tests.EditMode  (Assets/Tests/EditMode/)
├── Unit tests for Core and Systems (pure C#, no Unity runtime)
├── References: Core, Systems, MessagePipe, NUnit
└── Platform: Edit Mode only

KlondikeSolitaire.Tests.PlayMode  (Assets/Tests/PlayMode/)
├── Integration tests requiring Unity runtime
├── References: Core, Systems, Views, VContainer, Unity.TestFramework
└── Platform: Play Mode

KlondikeSolitaire.Editor          (Assets/Editor/)
├── Editor-only tools (CardAtlasGenerator)
├── References: Core, UnityEditor
└── Platform: Editor only (excluded from builds automatically)
```

**Dependency direction enforced at compile time:**
```
Views → Systems → Core
         ↓
    MessagePipe (messages defined in Core, published/subscribed in Systems & Views)

Editor → Core (Editor-only, never in builds)
```

## 7. Core Infrastructure Systems

### 7.1 ReactiveProperty\<T\>

**Purpose:** Minimal observable property for model→view data binding. ~50 lines.

**Behavior:**
- Holds a value of type `T`
- On set: compares with `EqualityComparer<T>.Default`. If changed, notifies all subscribers
- `Subscribe(Action<T>)` returns `IDisposable` for cleanup
- Views call `model.Property.Subscribe(handler).AddTo(compositeDisposable)`

**Why custom, not R3/UniRx:** 5 observable properties don't justify a library dependency. A 50-line class is auditable and allocation-free.

**CompositeDisposable:** Collects `IDisposable` subscriptions. Views create one, call `Dispose()` in `OnDestroy`. ~30 lines.

### 7.2 MessagePipe Configuration

**Purpose:** Cross-system and system→view event bus.

All messages are `readonly struct` (zero allocation). Registered in GameLifetimeScope.

| Message | Published By | Subscribed By | Payload |
|---------|-------------|---------------|---------|
| CardMovedMessage | MoveExecutionSystem | PileView, BoardView, HintSystem | sourcePileId, destPileId, cardCount |
| CardFlippedMessage | MoveExecutionSystem | CardView | pileId, cardIndex |
| ScoreChangedMessage | ScoringSystem | HudView | newScore, delta |
| GamePhaseChangedMessage | GameFlowSystem | HudView, OverlayView, InputView, BoardView | newPhase |
| AutoCompleteAvailableMessage | AutoCompleteSystem | HudView | isAvailable |
| UndoAvailabilityChangedMessage | UndoSystem | HudView | isAvailable |
| HintHighlightMessage | HintSystem | CardView, PileView | sourceCardIndex, sourcePileId, destPileIds[] |
| HintClearedMessage | HintSystem | CardView, PileView | (empty) |
| BoardStateChangedMessage | MoveExecutionSystem | HintSystem, NoMovesDetection, AutoComplete, **GameFlowSystem** | (empty — triggers re-evaluation including win check) |
| NoMovesDetectedMessage | NoMovesDetectionSystem | OverlayView | (empty) |
| WinDetectedMessage | GameFlowSystem | WinCascadeView, OverlayView | finalScore |
| NewGameRequestedMessage | HudView | GameFlowSystem | (empty) |

### 7.3 VContainer LifetimeScope

**Purpose:** Single GameLifetimeScope wires everything.

Registration pseudo code:
```
Configure(builder):
  // Models — singleton, shared state
  Register BoardModel as Singleton
  Register ScoreModel as Singleton
  Register GamePhaseModel as Singleton

  // Systems — singleton, injected with models + publishers/subscribers
  Register DealSystem as Singleton
  Register MoveValidationSystem as Singleton
  Register MoveExecutionSystem as Singleton, as IDisposable
  Register ScoringSystem as Singleton, as IDisposable
  Register UndoSystem as Singleton, as IDisposable
  Register HintSystem as Singleton, as IDisposable
  Register AutoCompleteSystem as Singleton, as IDisposable
  Register NoMovesDetectionSystem as Singleton, as IDisposable
  Register GameFlowSystem as Singleton, as IDisposable, as EntryPoint

  // MessagePipe — all message brokers
  RegisterMessagePipe options
  RegisterMessageBroker for each message type

  // Config SOs — serialized on GameLifetimeScope, registered as instances
  RegisterInstance LayoutConfig
  RegisterInstance AnimationConfig
  RegisterInstance ScoringConfig
  RegisterInstance InputConfig
  RegisterInstance CardSpriteMapping    (pre-populated by Editor tool — direct sprite refs)

  // Views — find in scene hierarchy
  RegisterComponentInHierarchy for each View type
```

**Why single scope:** No menus, no scene transitions, no sub-features needing child scopes. One scope is correct for a single-screen game.

## 8. Gameplay Systems

### 8.1 Card & Board Models

**CardModel** — identity + observable face state:
- `Suit` (readonly, set at construction)
- `Rank` (readonly, set at construction)
- `IsFaceUp` (ReactiveProperty\<bool\>) — CardView subscribes to update visuals
- Derived: `Color` (Red if Hearts/Diamonds, Black otherwise), `Value` (1-13 for Ace-King)

**PileModel** — ordered collection of cards:
- `PileType` (Stock, Waste, Foundation, Tableau)
- `PileIndex` (0-3 for foundations, 0-6 for tableau, 0 for stock/waste)
- `Cards` (List\<CardModel\>) — ordered bottom to top
- `TopCard` — peek at top card (or null if empty)
- `Count` — number of cards
- `FaceUpCount` — number of face-up cards (tableau only)

**BoardModel** — the complete game board:
- `Stock` (PileModel)
- `Waste` (PileModel)
- `Foundations` (PileModel[4])
- `Tableau` (PileModel[7])
- `GetPile(PileId)` — unified access by ID
- `AllPiles` — flat iteration over all 13 piles

**PileId** — struct identifying a pile:
- `PileType` + `Index` (e.g., Tableau:3, Foundation:1, Stock:0)
- Used as the universal pile reference in moves, messages, and undo commands

**ScoreModel:**
- `Score` (ReactiveProperty\<int\>) — HudView subscribes

**GamePhaseModel:**
- `Phase` (ReactiveProperty\<GamePhase\>) — Views subscribe to enable/disable based on phase
- Enum values: `Dealing`, `Playing`, `AutoCompleting`, `Won`, `NoMoves`

### 8.2 DealSystem

**Purpose:** Create a shuffled deck, deal to tableau and stock.

**MVS:** System only (no dedicated view — BoardView handles deal animation).

**Algorithm (pseudo code):**
```
CreateDeal():
  deck = create 52 CardModels (all face-down)
  shuffle deck using Fisher-Yates (System.Random)

  cardIndex = 0
  for col = 0 to 6:
    for row = 0 to col:
      add deck[cardIndex] to Tableau[col]
      cardIndex++
    flip Tableau[col].TopCard face-up

  add remaining cards (index 28-51) to Stock

  return DealResult (the board state)
```

**Events:** Publishes `DealCompletedMessage` after board is set up.

**Config:** None — dealing rules are fixed.

### 8.3 MoveValidationSystem

**Purpose:** Pure function — given a proposed move, return whether it's legal.

**MVS:** System only, stateless. Called by InputView, HintSystem, NoMovesDetection, and AutoComplete.

**Core method:** `IsValidMove(BoardModel board, PileId source, PileId dest, int cardCount) → bool`

**Validation rules (pseudo code):**
```
Tableau → Tableau:
  source cards must form valid descending alternating-color sequence
  if dest is empty: source bottom card must be King
  if dest is not empty: source bottom card must be opposite color AND rank = dest top rank - 1

Tableau → Foundation:
  only 1 card
  if dest is empty: card must be Ace, dest now tracks this suit
  if dest is not empty: card suit must match AND rank = dest top rank + 1

Waste → Tableau:
  same as Tableau→Tableau with 1 card from waste top

Waste → Foundation:
  same as Tableau→Foundation with waste top card

Foundation → Tableau:
  only 1 card (the top)
  follows standard Tableau placement rules (alternating color, descending rank)
```

**Auto-move target selection:** `FindBestTarget(BoardModel board, PileId source, int cardCount) → PileId?`

Priority order (matches GDD Section 5.4):
```
1. Foundation (if cardCount == 1 and card can go to matching foundation)
2. Tableau — leftmost valid column (iterate 0→6, return first match)
3. null (no valid target)
```

Used by InputView for single-tap auto-move. This is game logic (target priority rules), not input routing, so it belongs in the validation system.

**Architectural note:** Stateless. No side effects. Takes board as parameter — easy to test with any board configuration.

### 8.4 MoveExecutionSystem

**Purpose:** Execute a validated move — mutate board, trigger scoring, push undo command.

**MVS:** System. Mutates BoardModel, delegates to ScoringSystem and UndoSystem.

**Dependencies:** BoardModel, ScoringSystem, UndoSystem, IPublisher\<CardMovedMessage\>, IPublisher\<CardFlippedMessage\>, IPublisher\<BoardStateChangedMessage\>

**Core flow (pseudo code):**
```
ExecuteMove(PileId source, PileId dest, int cardCount):
  // Remove cards from source
  cards = source.RemoveTop(cardCount)

  // Add cards to destination
  dest.AddCards(cards)

  // Determine move type for scoring
  moveType = DetermineMoveType(source.PileType, dest.PileType)
  scoreDelta = ScoringSystem.CalculateScore(moveType)

  // Auto-flip: if source is Tableau and new top card is face-down, flip it
  flippedCard = false
  if source.PileType == Tableau AND source.TopCard != null AND !source.TopCard.IsFaceUp:
    source.TopCard.IsFaceUp.Value = true
    scoreDelta += ScoringSystem.CalculateScore(FlipCard)
    flippedCard = true
    publish CardFlippedMessage

  // Apply score
  ScoringSystem.ApplyDelta(scoreDelta)

  // Push undo
  UndoSystem.Push(MoveCommand(source, dest, cardCount, scoreDelta, flippedCard))

  // Notify
  publish CardMovedMessage(source, dest, cardCount)
  publish BoardStateChangedMessage
```

**Special moves:**
- `DrawFromStock()`: Move top stock card to waste, flip face-up. No score.
- `RecycleWaste()`: Move all waste cards to stock, flip all face-down. Maintain order. No score.

Both create undo commands and publish BoardStateChangedMessage.

### 8.5 ScoringSystem

**Purpose:** Calculate and apply score changes with clamping.

**MVS:** System. Mutates ScoreModel.

**Dependencies:** ScoreModel, ScoringConfig, IPublisher\<ScoreChangedMessage\>

**Score table (from GDD, stored in ScoringConfig SO):**

| Move Type | Points |
|-----------|--------|
| Waste → Tableau | +5 |
| Waste → Foundation | +10 |
| Tableau → Foundation | +10 |
| Foundation → Tableau | −15 |
| Flip face-down card | +5 |
| Draw from stock | 0 |
| Recycle waste | 0 |

**Logic:** `ApplyDelta(int delta)` — adds delta, clamps to 0, publishes ScoreChangedMessage.

### 8.6 UndoSystem

**Purpose:** Record all game actions, reverse them on demand.

**MVS:** System. Owns a Stack\<MoveCommand\>.

**Dependencies:** BoardModel, ScoringSystem, IPublisher\<UndoAvailabilityChangedMessage\>, IPublisher\<BoardStateChangedMessage\>

**MoveCommand (readonly struct):**
```
MoveType (Normal, DrawFromStock, RecycleWaste)
SourcePileId
DestPileId
CardCount
ScoreDelta
WasCardFlipped (bool — did auto-flip happen on source after move?)
WasteCardCount (only for RecycleWaste — how many cards to move back)
```

**Undo logic (pseudo code):**
```
Undo():
  command = stack.Pop()

  switch command.MoveType:
    Normal:
      // Move cards back
      cards = dest.RemoveTop(command.CardCount)
      source.AddCards(cards)
      // Un-flip if a card was auto-flipped
      if command.WasCardFlipped:
        source.TopCard.IsFaceUp.Value = false
      // Reverse score
      ScoringSystem.ApplyDelta(-command.ScoreDelta)

    DrawFromStock:
      // Move waste top back to stock, flip face-down
      card = waste.RemoveTop(1)
      card.IsFaceUp.Value = false
      stock.AddCards(card)

    RecycleWaste:
      // Move stock cards back to waste, flip face-up, restore order
      cards = stock.RemoveAll()
      reverse cards
      for each card: card.IsFaceUp.Value = true
      waste.AddCards(cards)

  publish BoardStateChangedMessage
  publish UndoAvailabilityChangedMessage(stack.Count > 0)
```

**Clear:** On new game, stack is cleared.

### 8.7 MoveEnumerator (Shared Utility)

**Purpose:** Enumerate valid moves on the board. Used by both HintSystem and NoMovesDetectionSystem to avoid duplicate logic.

**MVS:** Stateless utility in the Systems assembly. Not a system — no subscriptions, no state.

**Dependencies:** BoardModel (passed as parameter), MoveValidationSystem (passed as parameter)

**Methods:**
```
// Collect all valid moves into a provided list (avoids allocation on repeated calls)
EnumerateAllValidMoves(BoardModel board, MoveValidationSystem validation, List<Move> results):
  results.Clear()
  // Waste → Foundation (highest priority)
  for each foundation:
    if IsValidMove(waste, foundation, 1): add move

  // Waste → Tableau
  for each tableau column:
    if IsValidMove(waste, column, 1): add move

  // Tableau → Foundation
  for each tableau column:
    if column.TopCard exists AND IsValidMove(column, foundation, 1): add move

  // Tableau → Tableau (each valid sub-sequence)
  for each source column:
    for each valid face-up sub-sequence starting point:
      for each dest column:
        if IsValidMove(source, dest, subSequenceCount): add move

  // Stock draw (if stock has cards)
  if stock.Count > 0: add DrawFromStock move

// Short-circuit: return true on first valid move found
HasAnyValidMove(BoardModel board, MoveValidationSystem validation) → bool:
  // Same enumeration order as above, but returns true immediately on first match
  // Returns false only after exhausting all possibilities
```

### 8.8 HintSystem (uses MoveEnumerator)

**Purpose:** Cache valid moves, cycle through them on hint button press.

**MVS:** System. Reads BoardModel (no mutation).

**Dependencies:** BoardModel, MoveValidationSystem, MoveEnumerator, ISubscriber\<BoardStateChangedMessage\>, IPublisher\<HintHighlightMessage\>, IPublisher\<HintClearedMessage\>

**Algorithm (pseudo code):**
```
// Called when board state changes — invalidate cached hints
OnBoardStateChanged():
  _cachedMoves = null
  _hintIndex = -1
  publish HintClearedMessage

// Called when hint button pressed
GetNextHint():
  if _cachedMoves == null:
    _cachedMoves = new List<Move>()
    MoveEnumerator.EnumerateAllValidMoves(board, validation, _cachedMoves)
  if _cachedMoves is empty: return  // no-op
  _hintIndex = (_hintIndex + 1) % _cachedMoves.Count
  publish HintHighlightMessage(_cachedMoves[_hintIndex])
```

**Cycle reset:** Board state change via `BoardStateChangedMessage` clears cache and index.

### 8.9 AutoCompleteSystem

**Purpose:** Detect when the game is trivially winnable, generate the move sequence.

**MVS:** System. Reads BoardModel.

**Dependencies:** BoardModel, ISubscriber\<BoardStateChangedMessage\>, IPublisher\<AutoCompleteAvailableMessage\>

**Detection (called on every BoardStateChangedMessage):**
```
IsAutoCompletePossible():
  if stock.Count > 0: return false
  if waste.Count > 0: return false
  for each tableau column:
    for each card in column:
      if !card.IsFaceUp: return false
  return true
```

**Sequence generation:**
```
GenerateMoveSequence():
  moves = []
  repeat:
    lowestMove = null
    lowestRank = MAX

    for each tableau column:
      if column is empty: continue
      topCard = column.TopCard
      targetFoundation = foundation matching topCard.Suit
      if IsValidMove(column, targetFoundation, 1):
        if topCard.Rank.Value < lowestRank:
          lowestRank = topCard.Rank.Value
          lowestMove = (column → targetFoundation)

    if lowestMove == null: break  // done
    moves.add(lowestMove)
    // Simulate the move on a copy or execute immediately
  return moves
```

**Architectural note:** The System generates the sequence. The View (BoardView) iterates it asynchronously with animation delays using UniTask + PrimeTween.

**Undo interaction:** Auto-complete moves do NOT push to the undo stack. Once auto-complete starts, the game is trivially won (all cards face-up, no stock/waste). Undo is disabled during AutoCompleting phase, and Won is a terminal state. The undo stack is cleared on the subsequent New Game.

### 8.10 NoMovesDetectionSystem

**Purpose:** After each move, check if any valid moves remain.

**MVS:** System. Reads BoardModel.

**Dependencies:** BoardModel, MoveValidationSystem, MoveEnumerator, GamePhaseModel, ISubscriber\<BoardStateChangedMessage\>, IPublisher\<NoMovesDetectedMessage\>

**Algorithm:**
```
OnBoardStateChanged():
  if GamePhase != Playing: return
  if MoveEnumerator.HasAnyValidMove(board, validation): return
  publish NoMovesDetectedMessage
```

Uses `MoveEnumerator.HasAnyValidMove()` — same enumeration order as HintSystem but short-circuits on first valid move found. No need to collect all moves.

### 8.11 GameFlowSystem

**Purpose:** Top-level game state machine. Orchestrates phase transitions.

**MVS:** System. Mutates GamePhaseModel.

**Dependencies:** GamePhaseModel, DealSystem, BoardModel, ScoreModel, UndoSystem, HintSystem, ISubscriber\<BoardStateChangedMessage\>, ISubscriber\<NoMovesDetectedMessage\>, ISubscriber\<NewGameRequestedMessage\>, IPublisher\<GamePhaseChangedMessage\>, IPublisher\<WinDetectedMessage\>, IPublisher\<DealCompletedMessage\>

**State machine:**
```
         ┌─────────────────────────────────────┐
         ▼                                     │
     [Dealing] ──deal done──► [Playing] ◄──────┘
                                │    │     (New Game from any state)
                   ┌────────────┘    └────────────┐
                   ▼                              ▼
           [AutoCompleting]              [NoMoves]
                   │                        (terminal)
                   ▼
               [Won] ◄── also from Playing
                   (terminal — no undo from Won)
```

**Note — GDD departure:** The GDD states "unlimited undo, all the way back to the initial deal." This TDD intentionally treats Won and NoMoves as terminal states. Once the game ends (win or no moves), only New Game is available. Auto-complete only triggers when all cards are face-up with empty stock/waste — the game is trivially won, so undo is meaningless. This simplifies the state machine and avoids edge cases (undoing the win cascade, undoing from the win overlay).

**Transitions:**
- `Dealing → Playing`: After DealSystem completes
- `Playing → AutoCompleting`: User calls `GameFlowSystem.StartAutoComplete()` (via HudView)
- `Playing → NoMoves`: NoMovesDetectedMessage received
- `AutoCompleting → Won`: All foundations have 13 cards each (checked via BoardStateChangedMessage)
- `Playing → Won`: Last card placed on foundation manually (checked via BoardStateChangedMessage)
- `Any → Dealing`: NewGameRequestedMessage received

**Win check** — triggered by `BoardStateChangedMessage` subscription (runs after every move):
```
OnBoardStateChanged():
  if Phase != Playing AND Phase != AutoCompleting: return
  if !CheckWin(): return
  Phase = Won
  publish WinDetectedMessage(ScoreModel.Score.Value)
  publish GamePhaseChangedMessage(Won)

CheckWin():
  for each foundation:
    if foundation.Count != 13: return false
  return true
```

### 8.12 InputView ↔ DragView Protocol

**Purpose:** Define the exact interaction between InputView (input detection) and DragView (visual drag management).

**Ownership:**
- **InputView** owns input detection: pointer down, move, up, tap, double-tap
- **DragView** owns drag visuals: card position during drag, sorting order elevation, snap-back animation
- **Neither** owns game logic — both call Systems for validation and execution

**Drag Lifecycle:**

```
1. POINTER DOWN (InputView)
   ├── Raycast at pointer position (Physics2D.OverlapPointAll)
   ├── Sort hits by sorting order → select topmost CardView
   ├── Determine draggable card count:
   │   ├── If Waste/Foundation top card: count = 1
   │   ├── If Tableau: count = face-up sequence from clicked card to pile top
   │   └── If not draggable (face-down, empty): ignore
   ├── Start drag timer — only begin drag after pointer moves > InputConfig.dragThreshold pixels
   └── Cache: originPileView, cardViews[], originPositions[]

2. DRAG THRESHOLD MET (InputView → DragView)
   ├── InputView calls DragView.BeginDrag(cardViews[], originPileView)
   ├── DragView:
   │   ├── Sets each card's sorting layer to "Drag" (above all other cards)
   │   ├── Stores originPositions for snap-back
   │   └── Enters dragging state
   └── InputView enters dragging mode (suppresses tap detection)

3. POINTER MOVE (InputView → DragView, every frame while dragging)
   ├── InputView reads pointer position from Input System
   └── Calls DragView.UpdateDragPosition(worldPosition)
       └── DragView sets card transforms to follow pointer with original offsets preserved

4. POINTER UP (InputView → DragView → Systems)
   ├── InputView calls DragView.EndDrag() → returns drop world position
   ├── InputView raycasts at drop position to find target PileView
   ├── InputView calls MoveValidationSystem.IsValidMove(source, dest, cardCount)
   │
   ├── IF VALID:
   │   ├── InputView calls MoveExecutionSystem.ExecuteMove(source, dest, cardCount)
   │   ├── DragView.CompleteDrag() — animates cards to target PileView positions (PrimeTween snap)
   │   └── DragView resets sorting layers to "Cards"
   │
   └── IF INVALID:
       ├── DragView.CancelDrag() — animates cards back to originPositions (PrimeTween + shake)
       └── DragView resets sorting layers to "Cards"
```

**Tap/Double-Tap (InputView only — DragView not involved):**
```
POINTER UP without exceeding drag threshold:
  ├── Single tap: InputView calls MoveValidationSystem.FindBestTarget(board, sourcePile, 1)
  │   ├── If target found: call MoveExecutionSystem.ExecuteMove + animate via CardAnimator
  │   └── If no target: no-op (or shake animation via CardAnimator)
  │
  └── Double tap (within InputConfig.doubleTapWindow):
      ├── Check if card can go to any Foundation
      ├── If yes: call MoveExecutionSystem.ExecuteMove + animate to foundation
      └── If no: no-op

Stock tap: InputView calls MoveExecutionSystem.DrawFromStock()
Empty stock tap: InputView calls MoveExecutionSystem.RecycleWaste()
```

### Card Prefab Structure

```
Card (root — FLAT, no children)
├── CardView (component — manages sprite swap, sorting, highlight)
├── SpriteRenderer (single — sprite set from Cards.spriteatlas via CardSpriteMapping)
└── BoxCollider2D (for input raycasting)
```

**This is intentionally minimal.** The multi-sprite approach (5-6 child SpriteRenderers per card) was rejected because:
1. **Overdraw**: 5-6 full-size opaque quads per card × stacked columns = catastrophic overdraw (see reference screenshot)
2. **Draw calls**: Different sorting orders fragment batching. 52 cards × 5-6 layers = 60-75+ draw calls
3. **Complexity**: Managing 6 child transforms, toggling, sorting per child

**Instead: Editor-time pre-composited single sprite.**
- `CardAtlasGenerator` (Editor tool) composites all 52 card faces into individual PNGs
- PNGs are packed into `Cards.spriteatlas` by Unity (per-platform sizing and compression)
- Each card has 1 SpriteRenderer. Sprite swapped between face/back/backStrip
- All cards share the same atlas texture → same material → **1 draw call for all cards**
- **Zero runtime compositing cost**

**Face-down:** `SpriteRenderer.sprite = backSprite` (full back) or `backStripSprite` (cropped strip, see overdraw section)
**Face-up:** `SpriteRenderer.sprite = cardSpriteMapping.GetFaceSprite(suit, rank)`
**Flip animation:** PrimeTween scales X: 1→0 (swap sprite at midpoint) → 0→1. Single SpriteRenderer, no child toggling.

### Sorting Layers (NON-NEGOTIABLE)

All SpriteRenderers must use these sorting layers. Define in Unity via Edit → Project Settings → Tags and Sorting Layers:

| Sorting Layer | Order | Usage | Sorting Order Range |
|---------------|-------|-------|-------------------|
| Background | 0 | Green felt background sprite | 0 |
| Board | 1 | Pile base outlines (13 total) | 0 |
| Cards | 2 | All card sprites during normal play | 0–100 (per-card, managed by PileView based on position in pile) |
| Drag | 3 | Cards being dragged (elevated above everything) | 0–20 (preserves relative order of multi-card drag) |
| Cascade | 4 | Win cascade bouncing cards and trail stamps | 0–200 (dynamic, managed by WinCascadeView) |

**Rules:**
- PileView assigns sorting order within the Cards layer based on card position (bottom of pile = low, top = high)
- DragView switches cards to Drag layer on `BeginDrag`, back to Cards layer on `CompleteDrag`/`CancelDrag`
- WinCascadeView uses the Cascade layer for the celebration animation
- Never use `renderer.sortingLayerName` strings — cache sorting layer IDs as `static readonly int`

- **Orthographic** at size appropriate for viewport (adapts to screen resolution via LayoutConfig)
- **Clear to solid color** (dark green, the felt table)

## 13. Rendering & GPU Strategy

### The Problem: Multi-Sprite Cards = Overdraw Catastrophe

The naive approach (5-6 SpriteRenderers per card) causes two critical problems:

1. **Overdraw multiplication**: Each card renders 5-6 full-size opaque quads stacked on top of each other. When cards overlap in tableau columns, overdraw compounds. A column of 7 stacked cards × 6 layers = **42x overdraw** at the bottom pixel. The reference screenshot shows this exact problem — colors range from blue (2x) to magenta (10x+).

2. **Draw call fragmentation**: Different sorting orders between overlapping cards break sprite batching. 52 cards × 5-6 SpriteRenderers = 60-75+ draw calls.

### The Solution: Editor-Time Card Texture Compositing + SpriteAtlas

**Core idea: 1 card = 1 SpriteRenderer = 1 quad. All cards share 1 SpriteAtlas = 1 material = 1 draw call.**

#### CardAtlasGenerator (Editor-only tool)

**Location:** `Assets/Editor/CardAtlasGenerator.cs` — `[MenuItem("Tools/Klondike/Generate Card Atlas")]`

```
Editor tool (runs once when art changes, ~2s):

1. Read source component sprites from CardSpriteMapping SO:
   ├── card_front.png (face background)
   ├── Rank sprites (A, 2-10, J, Q, K) from cards/card numbers/new/
   ├── Suit sprites (hearts, diamonds, flowers=clubs, spades) from cards/semi/new/
   ├── Figure sprites (jack, queen, re=king) from cards/figures/red/ and cards/figures/black/
   ├── card_back.png
   └── card_base.png

2. For each of 52 card faces:
   ├── Create temporary RenderTexture (card-sized, 512×716 for high-res source)
   ├── Blit card_front.png as base layer
   ├── Blit rank sprite at top-left position (and rotated at bottom-right)
   ├── Blit suit sprite below rank (and rotated at bottom-right)
   ├── Blit figure sprite centered (J/Q/K only — red figures for Hearts/Diamonds, black for Clubs/Spades)
   ├── ReadPixels into Texture2D
   └── Save as PNG to Assets/Art/Sprites/cards/generated/card_{suit}_{rank}.png

3. Also generate:
   ├── card_back_strip.png (top ~20% of card_back.png — for face-down overdraw reduction)
   └── (card_back.png and card_base.png used as-is from source)

4. Auto-populate CardSpriteMapping SO:
   ├── Map each (Suit, Rank) → generated face sprite reference
   ├── Set backSprite → card_back.png
   ├── Set backStripSprite → card_back_strip.png (generated)
   └── Set baseSprite → card_base.png

5. AssetDatabase.Refresh() + mark CardSpriteMapping SO dirty
6. Log: "Generated 53 sprites. Repack Cards.spriteatlas."
```

**Source sprites** (ranks, suits, figures, front/back/base) are only used by the Editor tool — never rendered during gameplay.

**Cards.spriteatlas** is the ONLY texture used during gameplay. All 52 pre-composited card faces + back + strip + base are packed by Unity into this atlas → single material → all batched. Unity handles per-platform sizing and compression automatically.

### Draw Call Budget (Target: 5-10)

| Element | Draw Calls | Why |
|---------|-----------|-----|
| Background (green felt) | 1 | Different texture than cards |
| All pile bases (13) + all cards (52) + cascade stamps | **1** | Same Cards.spriteatlas texture → same material → URP batches all consecutive same-material sprites into 1 call |
| HUD Canvas (score text + buttons) | 2-3 | TMP text = 1 call (TMP shader), button images = 1-2 calls (UI atlas) |
| Overlay Canvas (when visible) | 1-2 | Only rendered during win/no-moves |
| **Total (gameplay)** | **4-6** | |
| **Total (win cascade)** | **4-6** | Stamps use same Cards.spriteatlas = same batch as cards |

**How 1 draw call for all cards works:**
- URP 2D Renderer sorts sprites by: Sorting Layer → Sorting Order → Sort Axis
- After sorting, consecutive sprites with the same material are batched
- All cards use sprites from the same Cards.spriteatlas → same texture → same material
- Even with different sorting orders (for overlap), no material change between cards = **no batch break**
- 13 pile bases + 52 cards + up to 150 cascade stamps = still 1 draw call

### Sprite Atlas Plan

| Atlas | Type | Contents | Used When |
|-------|------|----------|-----------|
| Cards.spriteatlas (existing, expanded) | Unity SpriteAtlas | 52 pre-composited card face PNGs + card back + card back strip + card base (from `cards/generated/` + source files) | **Gameplay** — all card rendering. Unity handles packing, compression, per-platform sizing |
| UI.spriteatlas (existing) | Unity SpriteAtlas | HUD icons, buttons | Gameplay — UI rendering |

### Material Sharing

- **All game-board sprites** (cards, pile bases, cascade stamps) share the Cards.spriteatlas material. **Never** access `renderer.material` — it clones and breaks batching
- **Hint highlight**: Use `SpriteRenderer.color` tinting (packed as vertex data, does NOT break batching). PrimeTween animates color for pulse effect. No MaterialPropertyBlock needed
- **UI elements** share the UI atlas material via Canvas batching
- **Background** uses a separate material (different texture)

### Batching Strategy

| Technique | Where Applied | Impact |
|-----------|---------------|--------|
| Cards.spriteatlas (single atlas texture) | All cards, pile bases, cascade stamps | **The key optimization.** 1 atlas = 1 texture = 1 material = 1 draw call for all game board elements |
| URP sprite batching | Consecutive same-material sprites batched automatically | Works across different sorting orders as long as material is uninterrupted |
| SRP Batcher | Enabled globally | Handles shader constant batching for remaining draw calls |
| SpriteRenderer.color | Hint highlights, cascade effects | Vertex-colored tinting preserves batching (unlike MaterialPropertyBlock) |
| Canvas batching | UI elements | Standard uGUI batching for HUD/overlays |

### Overdraw Strategy (NON-NEGOTIABLE)

The reference screenshot shows what happens without overdraw management. **Every layer of every card is a fill-rate cost**, and solitaire stacks cards by design.

#### Overdraw Sources & Mitigations

| Source | Without Mitigation | With Mitigation | How |
|--------|-------------------|-----------------|-----|
| Face-up card (self) | 5-6x (multi-sprite layers) | **1x** | Composited single sprite — one quad per card |
| Face-down stacking (tableau) | N × full card back per stack level | **N × 20% strip** | Face-down cards use `backStripSprite` (top 20% of card back). Only the topmost face-down card before the face-up section uses full `backSprite` |
| Face-up stacking (tableau) | N × 6 layers per card | **N × 1 quad** | Single composited sprite per card |
| Pile bases under cards | 13 full quads always rendered | 13 full quads | Acceptable — only 1x per base, occluded by cards above. Could disable when pile has cards but complexity not justified |
| Background | 1x everywhere | 1x | Single solid-color camera clear or 1 quad. No layered backgrounds |
| Win cascade | 52 cards + 150 stamps | Same | Temporary celebration, ~3-5 seconds. Acceptable higher overdraw |

#### Face-Down Strip Sprite — The Key Overdraw Optimization

In tableau columns, face-down cards overlap with a small vertical offset (only the top ~20% of each card is visible). Without optimization:

```
BEFORE (full back sprites):
┌─────────────┐ Card 1 (face-down) — full back rendered
│ ░░░░░░░░░░░ │ ← only this strip is visible
├─────────────┤ Card 2 (face-down) — full back rendered over Card 1
│ ░░░░░░░░░░░ │ ← only this strip is visible
├─────────────┤ Card 3 (face-down)
│ ░░░░░░░░░░░ │
├─────────────┤ Card 4 (face-up) — renders on top of all
│             │
│  [5 of ♥]  │
│             │
└─────────────┘

Bottom pixel: 4x overdraw (4 full-size quads stacked)
```

```
AFTER (strip sprites for face-down):
┌─────────────┐ Card 1 — strip sprite (top 20% only)
├─────────────┤ Card 2 — strip sprite
├─────────────┤ Card 3 — strip sprite
├─────────────┤ Card 4 (face-up) — full composited sprite
│             │
│  [5 of ♥]  │
│             │
└─────────────┘

Bottom pixel: 1x overdraw (only Card 4's quad reaches here)
```

**Rules for strip sprite assignment (managed by PileView):**
- Face-down cards that have another card on top of them → `backStripSprite`
- The last face-down card (directly under first face-up card) → `backSprite` (full, because the exposed area is larger at the transition)
- Face-up cards → always full composited face sprite
- Cards on stock/waste/foundation → always full sprite (no partial overlap)

#### Worst-Case Overdraw Analysis

Tableau column 7 (worst case): 6 face-down + 7 face-up cards:
- Face-down strip cards: only the strip region has overdraw, and strips don't overlap each other vertically
- Face-up cards: each is 1 quad. The bottom-most face-up card has all 7 face-up cards above it
- **Worst pixel overdraw: ~7-8x** (7 face-up cards + background) — down from **48x** with multi-sprite approach
- Acceptable for desktop and mobile — single composited sprite per card keeps fill rate manageable

#### Overdraw Verification

After implementation, verify with Unity Scene View → Overdraw visualization mode:
- Tableau columns should show blue/green (1-4x), not yellow/red
- No magenta (8x+) areas except possibly at win cascade peak
- Compare against reference screenshot — must be dramatically better

### 13.1 Developer Setup Steps (GPU Optimization)

#### Step 0: Generate Card Sprites (MUST DO FIRST)

1. Open Unity Editor
2. Menu → **Tools → Klondike → Generate Card Atlas**
3. Wait for completion log: "Generated 53 sprites. Repack Cards.spriteatlas."
4. Verify `Assets/Art/Sprites/cards/generated/` contains 52 face PNGs + 1 back strip PNG
5. Verify `CardSpriteMapping` SO has all sprite references populated (no missing/null entries)

**Re-run this whenever source card art changes.** The generated PNGs are committed to version control.

#### Step 1: Configure Cards SpriteAtlas

1. Select `Assets/Art/SpriteAtlases/Cards.spriteatlas` in Project window
2. In Inspector, "Objects for Packing" must include:
   - `Assets/Art/Sprites/cards/generated/` (the 52 composited faces + back strip)
   - `Assets/Art/Sprites/cards/card_back.png` (full card back)
   - `Assets/Art/Sprites/cards/card_base.png` (empty pile outline)
3. Settings:
   - Type: Master
   - Include in Build: checked
   - Allow Rotation: **OFF** (card sprites need consistent orientation)
   - Tight Packing: checked
4. **Platform Overrides** (the key multi-resolution feature):
   - Default: Max Texture Size **4096**, Format Automatic
   - Standalone: Max Texture Size **4096**, Format BC7
   - Android: Max Texture Size **2048**, Format ASTC 6×6
   - iOS: Max Texture Size **2048**, Format ASTC 6×6
5. Click "Pack Preview" — verify all sprites fit without overflow warnings

6. Select `Assets/Art/SpriteAtlases/UI.spriteatlas`
7. Verify "Objects for Packing" includes `Assets/Art/Sprites/options/` and `Assets/Art/Sprites/home/`
8. Settings: Max Texture Size 1024, same platform overrides pattern

#### Step 2: Texture Import Settings

1. Select all sprites in `Assets/Art/Sprites/cards/generated/`
2. In Inspector → Import Settings:
   - Texture Type: Sprite (2D and UI)
   - Sprite Mode: Single
   - Generate Mip Maps: **OFF** (2D game, no distance variation)
   - Read/Write Enabled: **OFF** (not needed — Unity reads from atlas at build time)
   - Compression: platform-dependent (handled by SpriteAtlas overrides)
3. Source sprites in `Assets/Art/Sprites/cards/` (non-generated) need:
   - Read/Write Enabled: **ON** (CardAtlasGenerator reads pixels in Editor)
   - These are NOT included in the build atlas — only used by the Editor tool
4. UI sprites (`options/`, `home/`): Read/Write OFF, compression via UI.spriteatlas overrides

#### Step 3: Verify URP Settings

1. Open `Assets/Settings/PC_RPAsset.asset`
2. Confirm SRP Batcher is enabled (Quality → SRP Batcher: ON)
3. Set MSAA to Disabled (2D game, no benefit, wastes fill rate)
4. Disable HDR (no benefit for 2D solitaire, saves bandwidth)
5. Verify 2D Renderer is assigned (not default 3D forward renderer)

#### Step 4: Camera Setup

1. Select Main Camera
2. Set Projection: Orthographic
3. Set Orthographic Size: calculate based on desired world-space card layout (likely 5-6)
4. Set Background Type: Solid Color → dark green (#35654d or similar felt color)
5. Remove or disable any post-processing volumes (no benefit for 2D solitaire)
6. Disable MSAA on the camera
7. Set Culling Mask: Everything

#### Step 5: Verify Draw Calls Post-Implementation

1. Open Window → Analysis → Frame Debugger
2. Play the game, pause during normal gameplay
3. Verify total draw calls are in the 5-10 range
4. If cards are NOT batching into 1 draw call: check that all card SpriteRenderers use sprites from Cards.spriteatlas (same texture)
5. Switch Scene View to Overdraw mode and verify no red/magenta areas in tableau columns

## 14. Testing Strategy

### Unit Tests (EditMode) — Pure C#

Every System gets a corresponding test class. Tests run instantly — no Unity runtime needed.

**Test structure:** Arrange-Act-Assert. One assertion per test method.

**TestBoardFactory:** Helper class that builds known board states:
```
TestBoardFactory.EmptyBoard()               — all piles empty
TestBoardFactory.FullTableauBoard()          — standard deal, all cards placed
TestBoardFactory.AlmostWonBoard()            — one card remaining to win
TestBoardFactory.NoMovesBoard()              — board with zero valid moves
TestBoardFactory.AutoCompletableBoard()      — all face-up, no stock/waste
TestBoardFactory.CustomBoard(setup)          — place specific cards in specific piles
```

| System | Target | Key Test Cases |
|--------|--------|----------------|

| MoveExecutionSystem | 90% | Move execution, auto-flip, scoring delegation, undo command creation |

### Integration Tests (PlayMode)

- Full game flow: deal → play moves → win

- Auto-complete end-to-end

### Mocking Strategy

Pure interfaces for cross-system boundaries. Constructor injection means tests pass mock implementations directly — no mocking framework needed.

```
// Test ScoringSystem in isolation
var model = new ScoreModel();
var config = TestScoringConfig.Default();  // creates config with known values
var publisher = new TestPublisher<ScoreChangedMessage>();
var sut = new ScoringSystem(model, config, publisher);

sut.ApplyDelta(10);
Assert.AreEqual(10, model.Score.Value);
```

For MessagePipe publishers in tests: create a simple `TestPublisher<T>` that captures published messages for assertion.

## 15. Design Patterns Summary

| System | Pattern(s) | Justification |
|--------|-----------|---------------|
| All Systems | MVS (Model-View-System) | Project standard. Separates state, logic, and rendering |
| UndoSystem | Command | Textbook undo — each move is an invertible command |
| GameFlowSystem | State Machine | Clear phase transitions with defined entry/exit conditions |
| MoveValidationSystem | Strategy (implicit) | Validation rules per pile type — selected by PileType enum |
| CardAtlasGenerator | Builder (Editor-only) | Composites card layers into single sprite PNGs — one-time Editor tool, zero runtime cost |
| CardView | Observer (ReactiveProperty) | View reacts to model.IsFaceUp changes, swaps single sprite from pre-composited atlas |
| HudView | Observer (MessagePipe) | Reacts to score/availability messages |
| Win cascade | Object Pool | Trail stamps recycled from a pre-warmed pool |
| Card lifecycle | Object Pool (fixed) | 52 cards created once, repositioned forever |
| All configs | ScriptableObject | Tunable data separated from code |
| All DI | Constructor Injection (VContainer) | Systems: constructors. Views: [Inject] Construct method |
| All cross-system comms | Publish/Subscribe (MessagePipe) | Decoupled, zero-alloc readonly struct messages |
